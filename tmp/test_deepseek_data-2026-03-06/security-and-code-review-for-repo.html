The Staff Engineer's Security & Sanity Review Prompt

You are a Staff Engineer conducting a final, critical pre-release audit. Your review has two primary goals, which you must pursue with equal rigor:

1. The Deep Security & Vulnerability Sweep Perform an exhaustive, A-to-Z security review of this codebase and its operational model. Assume it will be run by strangers in unknown environments. Your mandate is to uncover any vector for:

  • Filesystem Attacks: Path traversal, symlink following, race conditions, or unexpected file reads/writes/deletions outside of intended directories.
  • Information Leaks: Exposure of the developer's internal paths (e.g., /Users/secret-project), PII, API keys, git credentials, or internal network topography in logs, cache files, or final reports.
  • Supply Chain & Dependency Risks: Malicious or vulnerable dependencies, or reliance on external services (like specific LLM APIs) in a way that could be exploited.
  • "Agent Going Off the Rails": Scenarios where the script, when invoked by an agent (like Claude Code), could be tricked into destructive actions (e.g., deleting files, exposing data) through malicious input, malformed project structures, or symlink attacks.
  • Configuration & Secret Management: Flaws in how the .env file is read, how API keys are handled in memory or subprocesses, and whether any secrets could be inadvertently captured in logs or caches (like .dev-report-cache.md).

2. The Critical Eye for Code Quality & Scope Analyze the architecture and code with a ruthless eye for what can be simplified, cut, or refactored before launch. Ask "why" repeatedly.

  • Over-engineering: Is the complexity of the multi-phase, multi-model pipeline justified by the problem? What's the simplest thing that could possibly work? Can phases be merged or eliminated?
  • Sloppy Code & Spaghetti: Identify areas with poor error handling, unclear logic, inconsistent patterns, or "patches" that seem to fix a symptom without addressing the root cause. Point out technical debt that will hinder future development.
  • Missing the Big Picture: Does the current implementation fully solve the user's core problem, or are there fundamental gaps in the UX, error messaging, or failure modes that haven't been considered?

Your final output must be a single, consolidated report with two clear sections: ## 🔒 Security & Vulnerability Findings and ## 🔪 Scope & Code Quality Findings. Be direct, prioritize issues by severity (Critical, High, Medium, Low), and for each finding, provide a clear, actionable recommendation.

The Executed Review: Findings for dev-activity-report-skill

Consolidated Pre-Release Audit Report for dev-activity-report-skill

Reviewer: Staff Engineer (Simulated) Repo: ngallodev/dev-activity-report-skill Date: 2026-02-17 Version: As of commit on review date

🔒 Security & Vulnerability Findings

Critical Severity

  • Finding 1: Command Injection via git log arguments in phase1_runner.py.

* Description: The script constructs git log commands by concatenating arguments, some of which (like --since) are derived from user-controlled environment variables (REPORT<em>SINCE) or command-line flags. If an attacker can influence the .env file or the invocation arguments, they could inject malicious commands. For example, a value like 2026-01-01; rm -rf / in REPORT</em>SINCE could be catastrophic. The use of subprocess.run with shell=True is not evident, but even with a list argument, path injection is possible if user input isn't strictly validated. The run_pipeline.py passes --since directly to the phase1 subprocess. Recommendation: Immediately sanitize and validate all date inputs. Use a strict regex or a library like dateutil.parser to parse and re-format the date into a safe, predictable string (e.g., YYYY-MM-DD) before* inserting it into the command list. Never trust raw input from .env or CLI args for shell commands. Switch to using datetime objects and passing pre-validated, formatted strings.

  • Finding 2: Insecure Handling of API Keys in Subprocesses.

* Description: The pipeline uses the claude CLI, which likely reads API keys from its own configuration. However, environment variables (like ANTHROPIC<em>API</em>KEY) could be present. The run_pipeline.py script spawns subprocesses (claude, python for phases). If the parent environment is passed through (typical default), all API keys and secrets are exposed to child processes. A malicious or vulnerable third-party library imported in a later phase (e.g., the Haiku draft phase if it used an external SDK) could potentially steal these keys via environment variable sniffing. * Recommendation: Explicitly manage the subprocess environment. Create a clean environment dictionary for subprocesses that only contains explicitly allowed variables. Never blindly pass os.environ. For the claude CLI, rely on its own config file, not environment variables passed from the parent.

High Severity

  • Finding 3: Path Traversal in EXTRA<em>SCAN</em>DIRS and APPS_DIRS.

* Description: The configuration allows for multi-root scans via APPS<em>DIRS and EXTRA</em>SCAN_DIRS. While these are intended to be paths, there is no validation to prevent path traversal sequences like ../ or absolute paths that point to sensitive system directories (e.g., /etc, /var, /). An attacker with write access to the .env file could configure the tool to scan and potentially leak the contents of any file the user has access to by placing a malicious path there. The fingerprinting process then hashes file contents, which would effectively exfiltrate data into the cache and reports. Recommendation: Implement strict path validation and canonicalization. Resolve all input paths to their absolute, canonical form and then check if they are subdirectories of a set of allowed* base directories (which could be configured by the user in a more secure way). Warn loudly if a path falls outside of $HOME or a designated "safe" workspace.

  • Finding 4: PII/Path Leakage in Cache Files (.dev-report-cache.md and .phase1-cache.json).

Description: The fingerprint for non-git directories is a hash of file contents. However, the cache files themselves store the project name and paths* in plain text (e.g., ## project-name in .dev-report-cache.md). The .phase1-cache.json likely contains the full directory structure scanned. If a user's project path contains their real name (/Users/john.doe/...) or a client's name (/work/Project-X-Ray/...), this information is persisted in plain-text cache files. This creates a data leakage risk if these cache files are ever accidentally committed to a repo, backed up, or exposed. Recommendation: Never store raw, absolute paths in cache files. Store only the basename of the project and, if needed, a content-addressable identifier. Consider obfuscating or redacting the path component from the cache. The goal is to identify the content for caching, not the location*. The location should be re-evaluated on each scan from the config.

  • Finding 5: Symlink Following in Directory Traversal.

* Description: phase1_runner.py recursively walks directories. There is no mention of preventing symlink following. A malicious actor could place a symlink within a scanned project directory that points to a sensitive location like ~/.ssh/ or /etc/passwd. The script would then follow the link and include the contents of those sensitive files in its fingerprinting and analysis, leading to their inclusion in the JSON payload and potentially the final report. Recommendation: Use os.path.realpath on every directory entry before processing. If the resolved path is outside the intended scan root, skip it. Or, configure the walker to explicitly not follow symlinks (e.g., by checking os.path.islink). A configuration option to enable* following symlinks could be provided, but it must be False by default and clearly documented as a security risk.

Medium Severity

  • Finding 6: Information Leak via .not-my-work and .skip-for-now Markers.

* Description: While clever, the naming of these markers creates a minor infoleak. If a user works on a project for a client named "Acme" and marks it .skip-for-now, the mere existence of that file in a directory structure, if ever exposed, signals the presence and current status of work related to "Acme". A more generic, consistent marker file (e.g., .dev-report-ignore) with internal directives would be less revealing. * Recommendation: Use a single ignore file (e.g., .dev-report-ignore) with a simple syntax: skip, not-mine, forked, etc. This centralizes the logic and reduces the number of "signal" files scattered throughout the filesystem.

  • Finding 7: Lack of Input Validation on .forked-work-modified Inference.

* Description: The logic that infers contributions in a forked repo by using git log, git diff, and file modification times is a potential source of error and could be gamed. An attacker who can write files to a repo (e.g., via a malicious PR that gets merged) could create file modification times that cause the script to incorrectly attribute their work to the user. This is less a security vulnerability and more an integrity/accuracy issue, but it's an attack on the output's trustworthiness. * Recommendation: This feature should be clearly documented as a "best guess" and potentially require manual review (the --interactive flag is a good start). The algorithm's heuristics must be made robust and transparent. Consider only considering commits that are in the user's local reflog or have specific author signatures.

🔪 Scope & Code Quality Findings

High Severity (for Code Quality)

  • Finding 8: Severe Over-engineering for the Core Use Case.

* Description: The 5-phase pipeline (Phase 1, 1.5, 2, 2.5, 3) with model delegation (Haiku -> Haiku -> Sonnet -> GPT for cache) is a marvel of complexity, but it's solving a problem that doesn't exist for most users. The core task is: "Scan my git projects and write a summary." A single Python script using git commands and a well-crafted prompt to a single capable model (Sonnet) could achieve 95% of the value with 5% of the code and complexity. The benchmarks show Phase 2 (Sonnet) is the irreducible cost and bulk of the time. The rest is micro-optimization that adds enormous surface area for bugs, configuration errors, and security holes. The caching and skipping mechanisms are a solution to a self-inflicted performance problem. * Recommendation: Cut scope aggressively. Before release, create a "simplicity mode" or even a v2 branch. The primary path should be a single, well-documented script that does the following: 1) Find projects, 2) Gather essential git stats, 3) Send a single, well-crafted prompt to a single user-configured model, 4) Render the output. The current multi-phase system should be an advanced, optional "performance" mode, not the default. This aligns with the "big picture" – getting a useful report, not achieving 65% token savings on a $0.04 run.

  • Finding 9: Configuration Sprawl and Lack of Sensible Defaults.

* Description: The .env file is massive and intimidating for a new user. Variables like PHASE1<em>MODEL, PHASE2</em>MODEL, PHASE15<em>MODEL, PHASE3</em>MODEL require the user to understand the entire pipeline architecture just to configure it. ALLOWED<em>FILE</em>EXTS is a list of 14+ extensions. This is configuration debt. A user shouldn't need to know or care about "Phase 1.5". They just want a report. The SUBSCRIPTION_MODE flag is also a leaky abstraction – it's handling a billing detail that should be invisible to the user's workflow. Recommendation: Collapse configuration. A single MODEL variable could suffice. If multi-phase is kept, all those PHASE</em><em>MODEL vars should be hidden behind a single PIPELINE</em>PRESET (e.g., PIPELINE<em>PRESET=fast, =cheap, =accurate). ALLOWED</em>FILE_EXTS should be a curated default list based on common dev environments, not a user-editable list. If a user needs to customize it, they can, but the default should be smart.

Medium Severity

  • Finding 10: "Sloppy" Error Handling in Subprocess Calls.

* Description: The code relies heavily on calling external CLIs (claude, git). Throughout the pipeline, there is insufficient error handling for when these calls fail. What happens if git log fails in a repo? What if the claude CLI is not on PATH or returns a non-zero exit code? The code likely crashes or proceeds with incomplete data, potentially leading to a malformed report or a cryptic error message. * Recommendation: Every subprocess call must be wrapped in a try-except block, check the return code, and handle the failure gracefully. For a git failure in one repo, log the error, skip that repo, and continue. For a claude CLI failure, provide a clear, actionable error message: "Error: claude command not found or failed. Please ensure the Claude Code CLI is installed and authenticated."

  • Finding 11: The .forked-work-modified Pattern is a Clever Patch, Not a Solution.

* Description: This feature attempts to automatically solve the complex problem of authorship attribution in forks. While clever, it's a heuristic patch that will be wrong in many edge cases (e.g., upstream commits merged in, multiple contributors). It introduces significant complexity and a potential for incorrect reports without a clear, robust way for the user to verify or correct the inference without manually editing the cache files. * Recommendation: Rethink this feature. For a v1, it's safer to either require a manually created .forked-work file or to simply list the forked repo with a note: "Forked from X – manual summary required." The complexity of automatic inference is not worth the risk of generating misleading professional summaries.

  • Finding 12: Ambiguous Naming and Abstractions.

* Description: Terms like "Phase 1.5" and model names (gpt-5.1-codex-mini) are internal jargon that has leaked into the user's .env and file names. The report filenames include a precise timestamp (-YYYYMMDDTHHMMSSZ) which is overkill. The separation between phase1<em>runner.py, phase1</em>5<em>draft.py, and render</em>report.py creates a mental overhead for anyone trying to understand the flow. * Recommendation: Rename for clarity. data<em>gatherer.py, draft</em>generator.py, report_renderer.py. Use human-readable date formats for the final report file (e.g., -2026-02-17). Ensure the user-facing documentation and config use user-centric terms, not the system's internal phase names.

Final Recommendation

Do not release this in its current state. The security vulnerabilities, particularly the command injection and path traversal risks, are showstoppers. The architectural complexity is a liability that will make maintenance and security auditing a nightmare. The recommendation is to pause the release, refactor the core into a simple, secure, single-phase script, and then build the performance optimizations as a separate, clearly marked advanced feature. The "big picture" value of the tool is high, but its current implementation is too risky and over-engineered to be put in the hands of the public.
