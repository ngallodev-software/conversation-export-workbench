3. SQL (Advanced) – what they probably mean

“Advanced” on HackerRank usually means:

  • Complex joins (multiple tables).
  • Aggregations: GROUP BY, HAVING.
  • Window functions: ROW<em>NUMBER(), RANK(), DENSE</em>RANK(), SUM() OVER (...).
  • Subqueries and exists / in.
  • Maybe simple performance-awareness (avoid extra subqueries if possible).

Key patterns to practice

  1. Joins

SELECT b.Id, b.Title, a.Name AS AuthorName FROM Books b JOIN Authors a ON b.AuthorId = a.Id;

  1. Group + aggregate
  • “Find authors with more than 5 books”:

SELECT a.Id, a.Name, COUNT(*) AS BookCount FROM Authors a JOIN Books b ON b.AuthorId = a.Id GROUP BY a.Id, a.Name HAVING COUNT(*) > 5;

  1. Top-N per group (classic test question)
  • “Get the most recent book per author”:

SELECT * FROM (     SELECT         b.*,         ROW_NUMBER() OVER (PARTITION BY b.AuthorId ORDER BY b.PublishedDate DESC) AS rn     FROM Books b ) t WHERE t.rn = 1;

  1. Filtering with EXISTS

SELECT a.* FROM Authors a WHERE EXISTS (     SELECT 1     FROM Books b     WHERE b.AuthorId = a.Id       AND b.PublishedYear > 2020 );

If you’re rusty here, doing a handful of HackerRank SQL practice challenges will help a lot with speed and confidence.

A .NET (Basic) HackerRank test is not actually “easy”—it's usually fundamentals under time pressure. The goal is to verify that you can build or understand a small piece of .NET code without getting lost in syntax, language basics, or Web API patterns.

Here’s what you should expect very realistically:

✅ 1. C# Language Fundamentals

Expect questions or code tasks involving:

Core syntax

  • Declaring variables, classes, methods
  • Using var, types, namespaces

Control flow

if / else switch for / foreach while

Value vs Reference types

  • struct vs class
  • string immutability

Basic OOP

  • Classes, interfaces, inheritance
  • Constructors, properties, access modifiers (public, private, internal)

Basic data types & collections

List<T>, Dictionary<TKey,TValue>, arrays []

➡️ You should know how to iterate, add, remove, sort.

⚡ 2. LINQ Basics

Very common.

You may be asked to manipulate collections:

var result = books.Where(b => b.Year > 2020)                   .Select(b => b.Title)                   .ToList();

Know:

  • Where, Select, OrderBy, FirstOrDefault, Any, Count

You won’t get advanced queries like GroupJoin unless it's labeled Intermediate.

🌐 3. ASP.NET Web API Basics

For Basic tests, expect:

Knowing what controllers look like

[ApiController] [Route("api/[controller]")] public class BooksController : ControllerBase {     [HttpGet]     public IEnumerable<Book> Get() { ... } }

CRUD patterns

They want to see you can implement:

  • GET all
  • GET by id
  • POST new resource
  • DELETE by id

Simple example:

[HttpDelete("{id}")] public IActionResult Delete(int id) {     var deleted = _service.Delete(id);     return deleted ? NoContent() : NotFound(); }

You won't need:

  • Authentication
  • Middleware customization
  • EF Core migrations

🧠 4. Error handling & null-checks

This is almost always tested.

You should know safe patterns like:

if (item == null) return NotFound();

?? operator:

var name = author?.Name ?? "Unknown";

🧵 5. Basic async/await (very likely)

They want to know you understand async I/O:

public async Task<Book> GetAsync(int id) {     return await _repo.Get(id); }

You do NOT need to implement complex threading patterns.

🔧 6. Understanding Dependency Injection

Just at a conceptual level:

  • Services injected into controllers
  • Interfaces used for decoupling

Example:

public BooksController(IBookService service) {     _service = service; }

They won’t ask about reflection, containers, or lifetimes.

🔎 7. File / String / Array manipulation tasks

Sometimes the .NET portion is a coding problem, not a Web API.

Examples:

  • Reverse a string
  • Count frequency of items
  • Transform a list of objects using LINQ

These require:

  • foreach, Split, Contains, int.Parse, etc.

❌ What .NET Basic does NOT include

You generally will NOT see:

❌ Clean architecture patterns ❌ Domain-driven design ❌ Complex EF Core querying ❌ Message queues / microservices ❌ Multi-threading patterns
