is post-push a valid name for a .git/hooks file

ChatGPT 2026-02-28 22:53

No — post-push is not a valid Git hook name, so Git will ignore a file with that name in .git/hooks.

Git only recognizes a fixed set of hook filenames. There is no native hook that runs after git push completes locally.

✅ Hooks related to pushing (what actually exists)

1. pre-push ✅ (client-side)

This is the closest thing you want.

Runs before objects are pushed to the remote.

Location: 

.git/hooks/pre-push

Typical uses:

  • Run tests before allowing push
  • Linting
  • Prevent pushing secrets
  • Block pushes to protected branches

Example: 

#!/bin/sh echo "Running tests before push..." npm test || exit 1

If this script exits non-zero → push is cancelled.

2. Server-side push hooks (on the remote repo)

These run after or during push, but only on the remote server, not your local repo:

HookWhen it runs
pre-receiveBefore refs updated
updatePer branch update
post-receiveAfter push accepted ✅ closest to “post-push”
post-updateAfter refs updated

These live on the Git server, not your local .git/hooks.

❌ Why post-push doesn’t exist

Git’s design assumes:

  • Client validates before push (pre-push)
  • Server reacts after push (post-receive)

A local post-push hook would be unreliable because:

  • The client doesn’t fully control what the server accepts
  • Network failures could occur after local completion

So Git never added one.

✅ If you need post-push behavior locally

You have three practical options:

Option A — Wrap git push (most common workaround)

Create an alias:

git config --global alias.pushx '!f() { git push "$@" && ./after-push.sh; }; f'

Then use:

git pushx

Runs script only if push succeeds.

Option B — Use pre-push + background task

Start something asynchronously:

#!/bin/sh ( ./after-push.sh & )

(Not perfect — runs before confirmation.)

Option C — Use server post-receive (best for CI/CD)

If this is automation or deployment related, this is usually the correct architectural place.

✅ Quick summary

Hook nameValid?Runs where
pre-push✅local machine
post-push❌(ignored)
post-receive✅remote server

How to implement (clean + repo-portable)

Step 1: Put hooks in the repo using core.hooksPath

This avoids manually copying into .git/hooks for each clone.

In the repo: 

mkdir -p .githooks git config core.hooksPath .githooks

Commit the .githooks/ directory so everyone gets it.

Step 2: Create .githooks/post-commit

#!/usr/bin/env bash set -euo pipefail


JENKINS_URL="http://localhost:8080" JOB="app-tracker-proof-of-life" TOKEN="localtrigger123"


SHA="$(git rev-parse HEAD)" BRANCH="$(git rev-parse --abbrev-ref HEAD)" REPO_PATH="$(pwd)"


Trigger Jenkins to build this exact commit


curl -fsS -X POST \   "${JENKINSURL}/job/${JOB}/buildWithParameters?token=${TOKEN}&GITSHA=${SHA}&BRANCHNAME=${BRANCH}&REPOPATH=${REPO_PATH}" \   >/dev/null


echo "Triggered Jenkins for ${BRANCH}@${SHA:0:8}"

Make executable: 

chmod +x .githooks/post-commit

Step 3: Configure Jenkins Pipeline job to accept parameters

Job type: Pipeline

  • Use Pipeline script or Pipeline script from SCM — either works, but for your approach I’d do Pipeline script (simpler when building by SHA from arbitrary local paths).

Add parameters:

  • GIT_SHA (string)
  • REPO_PATH (string)
  • BRANCH_NAME (string)

Then in Jenkinsfile, checkout that SHA from the local repo path:

pipeline {   agent any   parameters {     string(name: 'GIT_SHA', defaultValue: '', description: 'Commit SHA to build')     string(name: 'REPO_PATH', defaultValue: '', description: 'Local repo path that contains the commit')     string(name: 'BRANCH_NAME', defaultValue: '', description: 'Branch name (informational)')   }   stages {     stage('Checkout') {       steps {         sh '''           set -e           rm -rf src           git clone "${REPO_PATH}" src           cd src           git checkout "${GIT_SHA}"         '''       }     }     stage('Build/Test') {       steps {         sh '''           set -e           cd src           # your build/test/proof-of-life steps here         '''       }     }   } }

This makes Jenkins immune to “branch moved” problems.

Does this avoid duplication?

Mostly, yes:

  • You keep one dev clone per developer (recommended).
  • Jenkins still does its normal thing: it checks out into its workspace. That’s “duplication,” but it’s standard and avoids corrupting your dev working tree.

Trying to make Jenkins build directly inside /lump/apps/app-tracker is possible, but it’s a foot-gun (workspace pollution, leftover artifacts, branch switching, half-finished changes, etc.).

When this approach is not enough

If you later want “build on push” from multiple machines (not all commits happening on the same Jenkins host), or you want a true server-side “every update to any branch triggers,” then the bare repo + post-receive design becomes the clean universal solution.

But for your current reality (“dev happens on the Debian server”), post-commit → Jenkins build by SHA is a very reasonable setup.
