Strong Interview Answer

“One system I designed that I think illustrates my approach well was a backend API platform for a large B2B e-commerce system.”

1. Context

At Rexel we had a nationwide electrical supply platform where customers needed to search a very large product catalog and place orders through the website and internal sales tools.

The system needed to support:

  • thousands of SKUs
  • complex product attributes
  • high search responsiveness
  • integrations with internal ordering systems
  • both customer-facing and internal applications.

So the goal was to build a scalable API-driven architecture that multiple frontends and internal tools could use.

2. Architecture

I designed the system around a .NET Core Web API service layer with several supporting components.

High level architecture:

Frontend (Vue.js)








v ASP.NET Core API Layer






---- Catalog Service


---- Order Processing Service


---- Search Service






v Elasticsearch + SQL Server

Key components included:

API Layer

  • ASP.NET Core REST APIs
  • authentication and request validation middleware
  • standardized error handling and logging

Search Infrastructure

  • Elasticsearch cluster for fast product search
  • indexing pipeline to sync catalog updates from SQL Server

Data Layer

  • SQL Server as system of record
  • Entity Framework Core for transactional operations

3. Key Technical Decisions

1. API-first architecture

Instead of tightly coupling UI to backend logic, I designed everything as REST APIs so multiple consumers could use it:

  • web frontend
  • internal sales tools
  • integrations with ERP systems

This made the system much easier to extend later.

2. Separate search from transactional data

Product search was the performance bottleneck.

So I introduced Elasticsearch as a read-optimized search index while SQL Server remained the transactional system.

That allowed:

  • millisecond search queries
  • faceted filtering
  • scalable indexing pipelines.

3. Asynchronous indexing pipeline

When product data changed in SQL Server, we pushed updates through a background worker that rebuilt Elasticsearch documents.

This avoided expensive synchronous indexing inside API calls.

4. Reliability and monitoring

We added:

  • structured logging
  • retry logic for indexing jobs
  • health checks and service monitoring

so operations could quickly identify issues.

4. Outcome

The system:

  • handled large-scale catalog search with sub-second response times
  • enabled multiple teams to integrate through the API layer
  • supported the nationwide customer base with 99.5% uptime. fileciteturn0file3

It also became the foundation for additional services like automated order workflows and internal CRM tools.

Interview Answer (Smaller System)

“A good example is an API service I designed to automate document-based order processing.”

Context

At one point we had a manual workflow where orders were sent in as PDFs via email and staff had to manually extract the data and enter it into the system.

The goal was to design a small service that could:

  • receive the documents
  • extract structured order data
  • validate it
  • automatically create orders in our system.

Architecture

I designed it as a small API service with a processing pipeline.

Incoming Email / Upload








v ASP.NET Core API








v Processing Service

v SQL Server

Key components:

API Layer

  • ASP.NET Core Web API endpoint
  • accepts file upload or message payload
  • validates request and stores the document

Processing Pipeline

  • background worker processes the file
  • parses the PDF
  • extracts structured fields (customer, items, quantities)

Validation Layer

  • verifies required fields
  • applies business rules
  • rejects malformed data

Persistence

  • SQL Server stores orders and document metadata.

Key Design Decisions

1. Asynchronous processing

Parsing PDFs can take time, so I designed it so the API:

  • accepts the request quickly
  • queues the document
  • processes it in a background worker.

This prevented slow API responses.

2. Clear separation of concerns

I split responsibilities into:

  • API layer
  • processing pipeline
  • data access

That made the service easier to test and maintain.

3. Idempotency and error handling

Documents sometimes arrived multiple times, so I used:

  • document hashes
  • duplicate detection
  • retry logic for failed processing.

Result

The system eliminated manual data entry and allowed orders to be created automatically from incoming documents.

It also gave us:

  • faster order turnaround
  • fewer data entry mistakes
  • a reusable API for other systems.

Interview Answer (Your Actual System)

“One smaller system I designed automated order intake from emails. Many customers would send orders via email, often with information in the body or attachments, and the process was very manual.”

Context

Customers frequently emailed orders in various formats — sometimes structured text in the email body, sometimes PDFs or spreadsheets attached.

The goal was to automate intake of those emails and turn them into structured tasks in our system.

Architecture

I designed a service that used a dedicated mailbox as a lightweight queue.

Customer Email








v Dedicated Inbox (queue)








v Polling Service (.NET)








+-- Parse email body       +-- Process attachments       +-- Extract structured data








v Order Processing API








v SQL Database

How it Worked

  1. Customers forwarded order emails to a dedicated internal mailbox.
  1. A .NET service polled the inbox at regular intervals.
  1. When a new email appeared it would:

- read the email body - download any attachments - extract relevant data (customer info, order details, etc.)

  1. The extracted data was then passed into the order processing API, which validated and persisted the order.

Key Design Decisions

Using the mailbox as a queue

Instead of introducing additional infrastructure, we used the email inbox as a natural queue because:

  • emails already arrived there
  • it guaranteed message persistence
  • operations staff could inspect failures easily.

Idempotent processing

Email systems sometimes resend messages, so the service stored message IDs and hashes to ensure the same email wasn't processed twice.

Attachment handling

The parser supported multiple formats:

  • plain text
  • PDF attachments
  • spreadsheet files.

Each format had a specific extraction pipeline.

Failure handling

If parsing failed, the message was:

  • logged
  • moved to a failure folder
  • flagged for manual review.

This ensured the automation never silently dropped orders.

Result

The system significantly reduced manual order entry and allowed customer emails to be converted into structured orders automatically while still allowing manual review for edge cases.
