✅ Implementing DeleteBookAsync with EF Core

You want to:

  1. Look up the book by id.
  2. If it doesn’t exist → return false.
  3. If it exists → remove it and save changes → return true.

public async Task<bool> DeleteBookAsync(int id) {     // Find the book by primary key     var book = await _context.Books.FindAsync(id);


if (book == null)     {         return false; // Not found     }


_context.Books.Remove(book);     await _context.SaveChangesAsync();


return true; }

That’s the backend method they’ll want you to write.

✅ Implementing the DELETE endpoint

We’ll:

  1. Call DeleteBookAsync(id) in the service.
  2. If it returns false → book not found → return NotFound().
  3. If it returns true → return NoContent() (standard for successful DELETE).

[HttpDelete("{id:int}")] public async Task<IActionResult> Delete(int id) {     var deleted = await _bookService.DeleteBookAsync(id);


if (!deleted)     {         return NotFound();      // 404 if book doesn't exist     }


return NoContent();         // 204 on successful delete }

Quick notes on the attributes

  • [ApiController] – enables automatic model validation, binding, etc.
  • [Route("api/[controller]")] – route = api/books (because class name is BooksController).
  • [HttpGet] – responds to GET /api/books
  • [HttpDelete("{id:int}")] – responds to DELETE /api/books/123 with an int route parameter.

5. Dependency Injection in Program.cs

This part is very likely to appear on the test.

Goal: register:

  • Controllers
  • DbContext
  • Your service (BookService)

Minimal hosting model (Program.cs)

using Microsoft.AspNetCore.Builder; using Microsoft.Extensions.DependencyInjection; using Microsoft.Extensions.Hosting; using Microsoft.EntityFrameworkCore;


var builder = WebApplication.CreateBuilder(args);


// 1. Add controllers (for attribute-routed controllers like BooksController) builder.Services.AddControllers();


// 2. Register DbContext with EF Core // On HackerRank, they often use InMemory DB so it just runs in the sandbox. builder.Services.AddDbContext<AppDbContext>(options =>     options.UseInMemoryDatabase("BooksDb"));


// 3. Register your service with DI builder.Services.AddScoped<IBookService, BookService>();


var app = builder.Build();


// 4. Configure middleware & endpoint routing app.MapControllers();


app.Run();

What they’re testing here

  • DI registration syntax:

- AddDbContext<AppDbContext>(...) - AddScoped<IBookService, BookService>()

  • Knowing lifetime keywords:

- AddScoped – one instance per HTTP request (most common for services using DbContext). - AddTransient – new instance every time it’s injected. - AddSingleton – one instance for entire app lifetime.

  • That you understand this flow:

Program.cs registers services ➜ Controller asks for services in constructor ➜ DI engine creates/injects them ➜ Controller calls service ➜ service uses DbContext to talk to DB.
