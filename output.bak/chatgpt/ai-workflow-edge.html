Portfolio Project Architecture: AI Workflow Reliability Framework

Core concept

A phase-based workflow engine where each phase:

  • takes an input artifact
  • produces an output artifact (typed + stored)
  • runs validators/evals
  • either passes, retries, or escalates to human
  • logs everything for audit + learning

Think: “CI/CD for LLM work.”

System boundaries (what you’re building)

You will ship:

  1. Workflow Orchestrator API (runs phases, routes decisions)
  2. Artifact Store + Audit Log (replayable history)
  3. Validation + Eval Harness (tests, scorecards, failure classes)
  4. Human Review UI (minimal: approve/reject/edit)
  5. Observability (cost/latency/quality over time)

You will NOT ship:

  • custom ML training
  • fancy agent swarms
  • perfect UX

High-level components

1) Orchestrator Service (API)

Responsibilities:

  • defines workflow templates (phases + rules)
  • executes runs
  • persists artifacts + results
  • enforces state machine (pass/retry/escalate)

Key design choice:

  • Explicit state machine per run:

- Created → PhaseRunning → PhaseSucceeded/Failed → Escalated → Completed

2) LLM Gateway (Provider abstraction)

Responsibilities:

  • unified interface across OpenAI/Anthropic/local models
  • captures request/response metadata (latency, tokens, cost estimate)
  • supports retry policies + fallbacks (model A → model B)

This is huge for demonstrating “production AI” thinking.

3) Artifact Store + Audit Trail

Artifacts are the “unit of truth.”

Store:

  • input artifact
  • phase prompts/config
  • raw model output
  • parsed/structured output
  • validation results
  • human edits (diff)
  • final artifact

Make everything replayable.

4) Validators (Deterministic gates)

Validators should be boring and strict:

  • JSON schema validation
  • required fields present
  • range checks
  • “must cite sources” (if applicable)
  • regex safety checks
  • domain rules

These produce:

  • ValidationResult { pass/fail, reasons[], severity, suggested_fix }

5) Eval Harness (Quality measurement)

This is what makes it portfolio-worthy.

Two modes:

  • Offline evals: run a test dataset through workflows and score
  • Online evals: sample production runs and score continuously

Scoring outputs:

  • correctness (task-specific)
  • format compliance
  • hallucination risk proxy (heuristics)
  • cost + latency
  • human escalation rate

6) Human Review UI (minimal)

One screen:

  • shows artifact + model output + validation failures
  • buttons: Approve / Reject / Edit
  • edits become new artifact version
  • system learns: “this pattern required human intervention”

Even a basic web UI is enough.

7) Observability / Metrics

Expose metrics endpoints + dashboards:

  • p50/p95 latency by model + phase
  • token usage + cost estimate by workflow
  • validation fail rate by rule
  • escalation rate by phase
  • retry counts
  • “top failure categories”

This is how you prove you’re not just “using AI.”

Data model (minimal but strong)

Tables (or collections)

Workflows

  • WorkflowId, Name, Version, DefinitionJson

WorkflowRuns

  • RunId, WorkflowId, Status, CreatedAt, CompletedAt

Phases

  • PhaseId, WorkflowId, Name, Order, ConfigJson

Artifacts

  • ArtifactId, RunId, PhaseId, Type, Version, ContentJson, RawText, CreatedAt

LLMCalls

  • CallId, RunId, PhaseId, Provider, Model, PromptHash, TokensIn, TokensOut, LatencyMs, CostEstimateUsd, ResponseRaw

ValidationResults

  • ResultId, ArtifactId, ValidatorName, Pass, Severity, ReasonsJson

HumanReviews

  • ReviewId, ArtifactId, Reviewer, Action, EditedContentJson, Notes, CreatedAt

This is a clean “auditability-first” schema.
