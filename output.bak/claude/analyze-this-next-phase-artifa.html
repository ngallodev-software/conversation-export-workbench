Here's what Section 15 resolves and why:

Stack decisions made (all researched, not guessed):

  • Python + FastAPI — LLM calls are I/O-bound. FastAPI's async-first design allows a single worker to handle many concurrent requests while waiting on external APIs, which is exactly this workload. Flask blocks on every request — wrong architecture for this.
  • PostgreSQL over SQLite — you need native ENUMs, JSONB, FK constraints, and ON CONFLICT upsert semantics for idempotency. SQLite's write operations can be slower due to file locking and sequential operations and doesn't support the constraint types required. PostgreSQL was the only real choice.
  • SQLAlchemy 2.0 async + asyncpg — standard FastAPI data layer, first-class async support.
  • Anthropic primary / OpenAI fallback — different providers, not just different models. Same-provider fallback doesn't protect against an outage.

Work done in Section 15:

  • Full executable DDL with enums, FKs, and circular FK resolution
  • Concrete Python validator interface (no ambiguity between engineers)
  • Pure-function DecisionEngine — trivially unit testable, no DB coupling
  • Idempotency via deterministic CallId + ON CONFLICT DO NOTHING — unique constraints or upsert logic are necessary to avoid duplication
  • Committed demo fixtures as Python dicts (not external files)
  • 4 explicit human gates before code starts

My honest read now: This is implementation-ready. The 4 human gates are real — don't skip them, especially gate 1 (DDL review) and gate 3 (fixture sign-off). Everything else can be built directly from this doc.
