Both ideas have merit, but I'm actually more intrigued by the bug injector concept. Here's why:

Bug Injector - More Realistic Practice

Why it's better for getting back in rhythm:

  • Most real .NET work involves debugging existing code, not greenfield
  • Forces you to read and understand unfamiliar codebases (critical skill)
  • More engaging than "build another TODO app"
  • Directly practices what you'll do in interviews and on the job

How it could work:

  1. Base Application: Start with a working .NET app (e.g., a simple e-commerce site, blog engine, or task manager)
  1. Bug Categories the injector could introduce:

- Null reference exceptions (classic .NET) - Race conditions / threading issues - Memory leaks - SQL injection vulnerabilities - Incorrect async/await usage - Off-by-one errors - Configuration issues - Performance problems (N+1 queries) - Authentication/authorization bypasses

  1. Difficulty Levels:

- Obvious: App crashes with stack trace - Subtle: Wrong calculation, edge case failure - Nasty: Intermittent issues, race conditions

  1. Features:

- Generate a "bug ticket" with reproduction steps - Option to get hints - Verify the fix with tests - Track which bug types you've solved
