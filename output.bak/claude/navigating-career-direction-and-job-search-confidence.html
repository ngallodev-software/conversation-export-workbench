Tool/function calling in LLMs:

The LLM doesn't just return text. It can say "I need to call function X with parameters Y" and your code executes that function, then feeds the result back to the LLM.

Example:

  • User: "What's the weather in Portland?"
  • LLM decides: "I need to call get_weather(city='Portland')"
  • Your code executes that function
  • LLM receives: {"temp": 45, "condition": "rainy"}
  • LLM responds: "It's 45°F and rainy in Portland"

Question: Is this what you understood, or did you think it was something else?

Orchestration logic vs microservices:

I may have overstated this. Let me be precise:

Microservices pattern: Service A calls Service B, handles B's response, maybe retries if B is down, routes to Service C based on B's output.

Agent orchestration: Agent decides "I need tool X," you execute tool X, agent sees result, decides "now I need tool Y," you execute tool Y.

The similarity: Both involve routing logic, error handling, and chaining operations based on intermediate results.

The difference: In microservices, YOU write the routing logic. In agents, the LLM decides the routing logic dynamically.

Question: Does that distinction make sense, or am I still being unclear?

Error handling for AI vs traditional code:

Traditional code errors:

  • Network timeout → retry
  • Invalid input → return error message
  • Database locked → wait and retry

AI-specific errors:

  • LLM returns garbage → How do you detect it? Retry with different prompt? Fail the workflow?
  • LLM hallucinates → Says it called a function that doesn't exist, or returns data in wrong format
  • Non-deterministic failures → Same input, different outputs. Which one is "correct"?
  • Rate limits → OpenAI cuts you off after X requests. Queue the work? Fail gracefully?

The hard part: You can't just check for HTTP 200. The API returns success but the content is wrong. You need validation logic specific to LLM outputs.

Question: Have you dealt with APIs that return 200 but garbage data? How did you handle it?

RAG (Retrieval Augmented Generation) - detailed:

The problem RAG solves: LLMs don't know your company's specific data. You need to give them context.

How it works:

  1. Indexing phase (done once or periodically):

- Take documents (PDFs, wikis, database dumps) - Split into chunks (usually 500-1000 tokens each) - Convert each chunk to a vector embedding (array of numbers representing semantic meaning) - Store vectors in a database (Pinecone, Weaviate, Postgres with pgvector)

  1. Query phase (every time user asks a question):

- User asks: "What's our refund policy?" - Convert question to vector embedding - Search vector DB for most similar chunks (this is "retrieval") - Send those chunks + user question to LLM - LLM generates answer using the retrieved context

What you need to understand:

  • Vector embeddings: Just a way to represent text as numbers so you can do semantic search. You don't need to understand the math, just that similar concepts cluster together.
  • Chunking strategy: How to split documents. Too small = loses context. Too large = LLM can't process.
  • Retrieval logic: Do you grab top 5 chunks? Top 10? How do you rank them?

What you DON'T need to know yet:

  • How embedding models work internally
  • Vector database internals
  • Advanced chunking strategies

Question: Have you ever built a search feature where "similar" matters more than exact matches? (e.g., fuzzy search, "people who searched X also searched Y")
