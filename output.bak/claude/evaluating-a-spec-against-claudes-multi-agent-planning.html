This is a genuinely interesting spec to think through critically. Let me give you an honest assessment.

What Forge actually is vs. Claude Code's multi-agent mode

Claude Code's orchestration already does most of what this spec describes — it plans, implements, runs tests, attempts repairs, and can spawn subagents. The honest difference Forge offers is narrower than the spec implies:

  • Deterministic branching convention (forge/<runId>-<slug>) — Claude Code doesn't enforce this
  • Structured run artifacts persisted to .forge/runs/ — Claude Code's session history isn't designed for this
  • Provider agnosticism — the IAgentProvider abstraction is the most genuinely novel piece; Claude Code is obviously locked to Claude
  • Whitelist-constrained command execution — Claude Code is more permissive by default

So the real value proposition is: reproducible, auditable, provider-agnostic runs with structured artifacts. That's a coherent thesis, but the spec doesn't lean into it hard enough. It reads more like "Claude Code but in .NET" than a tool with a distinct identity.

On the resume gap question — be direct with yourself

It helps, but only if you execute it with enough depth to talk about it for 20 minutes in an interview. A spec-only project doesn't move the needle. What would actually help:

  • Working CLI with at least the Claude provider implemented
  • Real artifact output you can show in a demo
  • Ideally, you've used it on itself — dogfooding is a great interview story

The risk is building just enough to list it on a resume but not enough to defend it confidently. Interviewers at senior level will probe hard on architectural decisions, tradeoffs you faced, and what you'd do differently.

Critical weaknesses in the current spec

The IAgentProvider interface is too thin. AgentTask and WorkspaceContext are doing enormous amounts of undeclared work — the interesting design is entirely in those types, and the spec handwaves them. Before writing code you'd want to nail down what a AgentTask actually contains for the planning step vs. the repair step, since those are very different payloads.

The repair loop is naive. One repair attempt with no loop limit, no strategy variation, no fallback — in practice this either succeeds trivially or fails permanently. A more interesting design would track why each repair failed and adjust the prompt strategy accordingly, which is where real value over "just ask Claude again" emerges.

The planning output is underspecified. A "strict JSON plan" that includes branch name, files to edit, and implementation steps sounds concrete, but implementation steps for an AI agent are just... more natural language. The spec doesn't address how structured the steps actually need to be to be actionable, or how to validate the plan before committing to implementation.

There's no context management strategy. Sending "relevant file contents + plan to provider" is the hardest part of the whole system — what's relevant, how do you stay within context limits, how do you handle large repos? This is entirely glossed over and it's where real implementations fall apart.

A sharper alternative angle

Rather than competing with Claude Code on its own turf, you could pivot the core identity to: a structured diff-review and commit pipeline that wraps any AI agent output into a human-auditable PR workflow. The value isn't in the AI planning — it's in what happens after the AI writes code.

Concretely: Forge's job is to take AI-generated changes (from any source — Claude Code, Copilot, raw API), validate them against a checklist (tests pass, no main branch modifications, diff size within bounds, no secrets committed), produce a structured review artifact, and stage a clean PR. You'd be building the guardrails and audit layer rather than reimplementing the agent loop.

This would be more defensible technically, more useful in enterprise contexts (which aligns with your .NET background), and more differentiated. It also maps naturally to things senior engineers care about: reproducibility, auditability, and safe integration of AI into existing workflows.

That framing also gives you a much better interview story: "I built a tool that treats AI-generated code changes as untrusted inputs and applies the same rigor we'd apply to any external contribution."

Bottom line

Worth building, but only if you commit to a working implementation that you've actually run against a real repo. The pivot toward an audit/guardrail layer is worth considering — it's more novel, more enterprise-relevant, and plays to your experience rather than trying to out-feature a tool Anthropic is actively investing in.

The core problem with Forge as specified

Three alternatives worth considering

1. AI Change Auditor / "Trust Gate"

The real anxiety in enterprise .NET shops isn't "can AI write code" — it's "how do I know what the AI actually changed and whether it's safe to ship." Build a tool that sits between any AI agent's output and your repo:

  • Analyzes diffs for common AI failure patterns (hallucinated dependencies, removed error handling, test deletions, hardcoded values)
  • Produces a structured risk report per change
  • Enforces configurable rules before a commit is allowed

This is greenfield, it's language-agnostic at the diff layer but you'd specialize it for .NET/C# first, and it solves a real pain that enterprises will actually pay for. The interview story is also crisp: "AI writes code but someone still has to be responsible for it — I built the tool that makes that review tractable."

2. Prompt Regression Suite for .NET Teams

Teams using GitHub Copilot or Claude Code in their .NET repos have no way to know if updating their AI tooling or prompts broke their typical workflows. Think of it like a test suite, but for AI behavior:

  • You define expected behaviors ("when I ask to add a repository method, it should follow our naming conventions and include XML docs")
  • The tool runs these against your actual codebase as fixtures
  • You get a pass/fail report when you upgrade models or change prompts

This is genuinely novel, it maps directly to your testing background, and it addresses something that's going to become a real problem as teams mature their AI usage. The tooling space here is almost empty right now.

3. .NET Architecture Linter with AI Explanation Layer

This is the most pragmatic option for resume purposes because the core is achievable solo in a few weeks. Build a Roslyn-based analyzer that enforces architectural rules (no direct DbContext in controllers, no cross-layer dependencies, CQRS handler conventions, etc.) and then uses an AI layer to explain violations in plain English with suggested fixes rather than just flagging them.

The differentiator isn't the linting — Roslyn analyzers exist. It's the AI explanation layer that contextualizes violations against your team's specific architecture, not generic best practices. You'd configure it with your architecture description and it uses that as context when explaining why something is wrong.

Option 3 gets you to a working demo fastest and lets you talk about Roslyn, static analysis, AI integration, and .NET architecture patterns all in one project — which is exactly the surface area a senior .NET role will probe. It's also genuinely useful to any .NET team, which matters for the "would you use this yourself" credibility test in interviews.

Great choice — this one is genuinely novel. Let me think through what an MVP actually looks like.

The Core Concept

You're building a behavioral test suite for AI coding assistants. Just like unit tests assert that your code behaves correctly, these "prompt tests" assert that your AI tooling behaves correctly given your codebase's conventions.

The key insight: as teams adopt AI tools, they're accumulating implicit expectations about AI behavior with zero way to verify them. You're making those expectations explicit and executable.

What a "Prompt Test" Actually Is

This is the most important design decision to nail. A prompt test has four parts:

Fixture — a snapshot of real code from your repo that provides context (a class, a file, a directory)

Prompt — the instruction as a developer would naturally type it

Assertions — what the response must contain, structurally or semantically

Metadata — which model, which tool, what conventions file it depends on

A concrete example in a hypothetical test file format:

name: "Repository method follows naming conventions" fixture: src/Data/UserRepository.cs prompt: "Add a method to get all active users by role" assertions:   - type: contains_method     name_pattern: "GetByAsync"   - type: hasxmldocs   - type: nodirectsql   - type: uses_type: "IQueryable"   - type: semantic     description: "Method should filter by IsActive = true" model: claude-sonnet-4-5 tags: [repository-pattern, naming, async]

The semantic assertion type is the interesting one — it uses a second AI call to evaluate whether the first response satisfied a natural-language expectation. Meta, but it works.

MVP Architecture

/src   /Core     PromptTest.cs          -- test definition model     TestResult.cs          -- pass/fail + details     AssertionResult.cs   /Assertions     IAssertion.cs     ContainsMethodAssertion.cs     HasXmlDocsAssertion.cs     SemanticAssertion.cs   -- AI-evaluated assertion     RegexAssertion.cs     NoPatternAssertion.cs   /Providers     ILlmProvider.cs     ClaudeProvider.cs     OpenAiProvider.cs   /Runner     TestRunner.cs          -- loads, executes, collects results     FixtureLoader.cs       -- resolves fixture paths   /Reporting     IReporter.cs     ConsoleReporter.cs     JsonReporter.cs   /Cli     Program.cs             -- top-level CLI entry     RunCommand.cs     InitCommand.cs

The CLI surface for MVP is just two commands:

dotnet-prompt-test run ./tests/         # run all tests dotnet-prompt-test run ./tests/ --tags repository-pattern dotnet-prompt-test init                 # scaffold a sample test file

The Runner Flow

Load .ptest.yaml files     ↓ Resolve fixtures (read actual source files)     ↓ Build prompt context (fixture + prompt + conventions if configured)     ↓ Call LLM provider     ↓ Run each assertion against the response     ↓ For semantic assertions → second LLM call to evaluate     ↓ Aggregate results → reporter

The key thing the runner does that makes this non-trivial: it builds a repeatable prompt context. Every run sends the same fixture content, same conventions, same system prompt scaffold. That reproducibility is what makes the results comparable across model versions.

Assertion Types for MVP

Start with five, in order of implementation difficulty:

contains_pattern — regex match against response. Simplest, covers a lot of ground.

no_pattern — negative regex. Catches things like "don't use Thread.Sleep" or "don't omit async."

contains_method — parses the response for a method signature matching a pattern. You can do this with regex for MVP rather than full Roslyn parsing.

compiles — extracts code blocks from the response, attempts dotnet build against a temp project. This is hard but high value. Save it for post-MVP or make it optional.

semantic — the crown jewel. System prompt is something like: "You are evaluating whether an AI coding assistant response satisfies the following requirement. Respond with PASS or FAIL and a one-sentence reason." Then feed it the requirement and the response. Surprisingly reliable.

The Conventions File

This is a feature that makes the tool feel real rather than toy. Teams can provide a conventions file that gets injected into every prompt context:

# MyProject Coding Conventions


    

  • All repository methods must be async and return Task<T>
    • 

  • Use IQueryable for filterable queries, IEnumerable for finalized results  
    • 

  • XML documentation required on all public methods
    • 

  • Never reference DbContext directly outside of Repository classes
    • 

  • Follow the naming pattern: Verb + Entity + Qualifier + Async
    •

You load this once at runner startup and prepend it to every fixture prompt. Now your tests are asserting against your conventions, not generic .NET style — which is the whole point.

The Interesting Technical Problems

Fixture isolation — how much context is enough? Sending a single file is often insufficient; the AI needs to understand interfaces, base classes, related types. For MVP, support a fixture_includes list of additional files. Long-term this becomes a dependency graph problem.

Response extraction — LLM responses contain markdown, explanation prose, and code blocks mixed together. You need a reliable code block extractor before any code-targeted assertions run. Use a simple fence parser for MVP, it handles 95% of cases.

Result stability — LLMs are non-deterministic, so a test might pass 9/10 runs. For MVP, just document this and run each test once. Post-MVP you add a runs: 3 option that requires N of M to pass.

Baseline capture — a killer feature you can add early: dotnet-prompt-test baseline ./tests/ runs everything and saves the raw responses. Future runs can diff against the baseline, showing you exactly how behavior changed when you upgraded your model. This is the feature that would make a team actually depend on this tool.

What Makes This a Great Resume Project

You can talk about it across multiple dimensions in an interview:

System design — provider abstraction, assertion pipeline, fixture resolution, context assembly. There are real architectural decisions here.

Testing philosophy — you understand what makes a good test (deterministic, isolated, meaningful failure messages). You applied that to a novel domain.

AI integration — you used AI to evaluate AI, you thought carefully about prompt engineering, you dealt with non-deterministic outputs in a test harness context.

Practical value — you can demo it against a real .NET repo in under 2 minutes. That's the difference between a project you list and a project you show.

First week: CLI scaffold, YAML test file loader, Claude provider, regex assertions, console reporter. Get one test running end-to-end.

Second week: semantic assertion, conventions file support, JSON reporter, baseline capture command.

Third week: polish, a real test suite for the tool itself (dogfooding), README with examples, a recorded demo.
