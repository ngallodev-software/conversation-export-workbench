Here's a clean REST API design for a todo app. REST follows a resource-oriented approach, so everything revolves around your todo items (and optionally lists).

Core endpoints

Todo Items

MethodPathDescription
GET/todosList all todos
POST/todosCreate a new todo
GET/todos/:idGet a specific todo
PATCH/todos/:idUpdate a todo (partial)
DELETE/todos/:idDelete a todo

Example: Create a todo

Request: 

POST /todos Content-Type: application/json


{   "title": "Buy groceries",   "due_date": "2025-06-10",   "priority": "medium" }

Response (201 Created): 

{   "id": "todo-abc123",   "title": "Buy groceries",   "completed": false,   "due_date": "2025-06-10",   "priority": "medium",   "created_at": "2025-06-06T15:01:00Z",   "updated_at": "2025-06-06T15:01:00Z" }

Example: Mark as complete

PATCH /todos/todo-abc123 Content-Type: application/json


{ "completed": true }

Useful query parameters for listing

GET /todos?completed=false        # Only incomplete todos GET /todos?priority=high          # Filter by priority GET /todos?sort=due_date&order=asc # Sort by due date GET /todos?page=2&limit=20        # Pagination

HTTP status codes to use

CodeWhen to use

204 No ContentSuccessful DELETE

Should I version my API? What's the best practice there?

Claude 2025-06-06 15:10

API versioning is a good habit, especially if you'll have external clients (mobile apps, third-party integrations). The short answer: yes, version it — and do it in the URL path.

The three common approaches

1. URL path (recommended)

/v1/todos /v2/todos

  • Explicit, easy to test in a browser or curl.
  • Clients can migrate gradually: keep /v1 running while /v2 rolls out.
  • Works everywhere (no special headers needed).

2. Query parameter

/todos?version=1

  • Less clean, easy to forget, harder to route in most frameworks.

3. Accept header

Accept: application/vnd.myapp.v2+json

  • Technically "most RESTful" per purists.
  • Hard to test manually, less transparent.

My recommendation for a new project

https://api.yourapp.com/v1/todos

Start with v1 from day one. It costs almost nothing upfront and saves you pain later.

When to bump versions

  • Don't bump for additive changes (new optional fields, new endpoints) — these are backward-compatible.
  • Do bump for breaking changes:

- Removing or renaming a field - Changing a field's type - Changing authentication requirements

Deprecation strategy

  1. Announce the deprecation date.
  2. Add a Deprecation response header to the old version.
  3. Keep the old version alive for a reasonable period (3–6 months is common).
  4. Shut it down once traffic drops off.

For a personal todo app you might never need v2, but the pattern costs nothing and is good muscle memory.
