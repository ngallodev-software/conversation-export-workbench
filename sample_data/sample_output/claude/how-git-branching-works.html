Git branches are one of Git's most powerful features. Here's how they work:

What is a branch?

A branch is simply a lightweight, movable pointer to a specific commit. The default branch is usually called main (or master in older repos). Every time you make a commit on a branch, the pointer automatically moves forward to the new commit.

Why use branches?

Branches let multiple lines of development exist in parallel without interfering with each other:

  • Feature development — build a new feature in isolation; only merge it when it's ready.
  • Bug fixes — patch a production bug without including half-finished feature work.
  • Experimentation — try a risky refactor; discard the branch if it doesn't work out.
  • Code review — reviewers can check out your branch and test it before it lands in main.

Basic workflow

# Create and switch to a new branch git checkout -b feature/add-search


... do work, make commits ...


Switch back to main


git checkout main


Merge your feature branch


git merge feature/add-search


Clean up


git branch -d feature/add-search

How merging works

When you merge two branches, Git finds their common ancestor commit and combines the changes. If there are no conflicts, it's automatic. If two branches changed the same lines, Git flags a merge conflict for you to resolve manually.

A typical team workflow

BranchPurpose
mainStable, production-ready code
developIntegration branch for features
feature/*Individual feature work
hotfix/*Emergency production fixes

This pattern (called Git Flow) keeps production code stable while allowing active development to happen in parallel.

What's the difference between merge and rebase?

Claude 2025-06-05 10:08

Both integrate changes from one branch into another, but they do it differently and leave different history.

Merge

git checkout main git merge feature/add-search

  • Creates a new merge commit that ties together the histories of both branches.
  • Preserves the full history — you can see exactly when and how branches diverged and converged.
  • Non-destructive: existing commits are never changed.

o---o---o---M   ← main (M is the merge commit)      \     /       o---o     ← feature branch

Rebase

git checkout feature/add-search git rebase main

  • Replays your feature commits on top of main, one by one, creating new commits.
  • Results in a linear history — looks like the feature was always built on top of the latest main.
  • Rewrites commit history — the original commits are replaced with new ones (different SHA hashes).

o---o---o---o'---o'   ← main + rebased feature commits

When to use which

ScenarioUse
Merging a completed feature into mainMerge — preserves context
Keeping your feature branch up to date with mainRebase — cleaner history
Public/shared branchesMerge — never rebase commits others have
Personal, local branchesEither; rebase gives a cleaner log

The golden rule

Never rebase commits that have been pushed to a shared branch. Rewriting history forces other developers to reconcile diverged histories, which is messy.

A common pattern: rebase your private feature branch to keep it current, then do a final merge (or squash merge) into main.
