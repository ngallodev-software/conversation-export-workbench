Memory leaks in Node.js are almost always caused by unintentional object references that prevent the garbage collector from freeing memory. Here's how to find and fix them:

Step 1: Confirm it's actually a leak

First, monitor heap usage over time:

// Add this to your server and watch the output setInterval(() => {   const { heapUsed, heapTotal } = process.memoryUsage();   console.log(Heap: ${Math.round(heapUsed / 1024 / 1024)}MB used / ${Math.round(heapTotal / 1024 / 1024)}MB total); }, 5000);

If heapUsed grows steadily and never comes back down, you have a leak.

Step 2: Take heap snapshots

Run your server with the inspector:

node --inspect server.js

Open Chrome → navigate to chrome://inspect → click Open dedicated DevTools for Node.

  1. Go to the Memory tab.
  2. Take a baseline heap snapshot.
  3. Trigger the leaky behavior (make requests, run the workflow).
  4. Take a second snapshot.
  5. Compare: look for objects that grew significantly.

Step 3: Common culprits

Event listener accumulation

// BAD — adds a new listener every time this runs function handleRequest(req, res) {   someEmitter.on('data', (chunk) => { / ... / }); }


// GOOD — register once, or use .once() someEmitter.on('data', handleData); // outside the request handler

Unbounded caches

// BAD — grows forever const cache = {}; app.get('/user/:id', (req, res) => {   cache[req.params.id] = fetchUser(req.params.id); // never evicted });


// GOOD — use a LRU cache with a size limit import LRU from 'lru-cache'; const cache = new LRU({ max: 500 });

Forgotten timers/intervals

// BAD — interval keeps a reference alive const timer = setInterval(doWork, 1000); // Missing: clearInterval(timer) when done


// GOOD const timer = setInterval(doWork, 1000); process.on('SIGTERM', () => clearInterval(timer));

Closures holding large objects

// The closure keeps 'largeData' alive even if only 'id' is needed function createHandler(largeData) {   const id = largeData.id; // extract only what you need   return () => console.log(id); // closure only holds 'id' }

Useful tools

  • clinic.js (npm i -g clinic) — clinic heap -- node server.js generates a flame graph
  • node-memwatch — emits events when heap grows unexpectedly
  • heapdump — write heap snapshots from inside the process on demand

I found it — I was adding an event listener inside a request handler without removing it. How do I fix it cleanly?

Claude 2025-06-08 08:15

Great find! There are three clean patterns depending on your situation:

Pattern 1: Use .once() for one-shot listeners

If you only need the listener to fire one time per request:

// Automatically removed after firing once someEmitter.once('data', (chunk) => {   res.send(chunk); });

Pattern 2: Remove the listener when done

If the listener needs to fire multiple times but only for the lifetime of a request:

const handler = (chunk) => {   // process chunk };


someEmitter.on('data', handler);


res.on('finish', () => {   someEmitter.off('data', handler); // clean up when response is done });


req.on('close', () => {   someEmitter.off('data', handler); // clean up if client disconnects early });

Pattern 3: Use AbortController (modern Node.js)

For complex scenarios with multiple listeners:

const ac = new AbortController(); const { signal } = ac;


// The { signal } option auto-removes the listener when aborted someEmitter.on('data', handler, { signal }); anotherEmitter.on('event', otherHandler, { signal });


res.on('finish', () => ac.abort()); // removes ALL listeners at once

Pattern 4: Move registration outside the handler

If the emitter and handler are both long-lived, register once at startup:

// At startup — registered once someEmitter.on('data', globalDataHandler);


// In your request handler — no listener registration here app.get('/stream', (req, res) => {   // just use the already-registered global handler });

Quick sanity check

Add this temporarily to catch future violations:

// Warns if any single emitter exceeds 10 listeners someEmitter.setMaxListeners(10); // Node.js will print a warning if you exceed this

Pattern 2 or 3 is usually the right choice for per-request listeners tied to a response lifecycle.
